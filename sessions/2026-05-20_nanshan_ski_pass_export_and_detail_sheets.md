## 学到的小知识

1. **雪票一单可多票要看店**：崇礼旗舰店 25-26 是 1:1（572 张/572 单），南山是 1.17（542 张/463 单，58 单多票）。同一聚合脚本崇礼时多票兜底逻辑（SUM/MIN/分号连接）不触发，南山触发但结果正确。后续做万龙/总部时不能假设崇礼的 1:1。

2. **`have_refund` 字段并非（1/0/NULL）三值**：DB 实际只有 1（已退）和 NULL（未退），没有 0。转标签时 `1 → "是" / NULL → "否"`，**不能用 `0 → 否"`** 否则全部映射成"是"。同类陷阱：其它 int 类型字段也可能"业务未写 0、只写 1 / NULL"。

3. **渠道订单号 `_ZF_NN` 是 snowmeet 支付单号约定**：年度雪票订单号（如 `QJ_XP_260405_00001`）+ `_ZF_NN` 后缀 = 支付单号（如 `..._ZF_02`）。同样约定适用于退款 `_TK_MM` / 分账 `_FZ_MM`（与 2026-05-16 支付流水 `out_trade_no` 命名一致）。外部七色米/自我游"雪票列表"导出的「渠道订单号」就是这个支付单号，匹配 snowmeet 订单号用 `split('_ZF_')[0]`。

4. **合并单元格场景整行上色不能漏明细列**：openpyxl 一对多展开 + 订单级列合并时，如果只给订单级列上底色、明细列保白，视觉上每个多明细订单的右半边是断的。要么整行（含明细列）一起上色，要么全留白；不能"只半边上色"。

5. **阈值"很低"和"较大"对齐成两端切分**：用户协作偏好—阈值有歧义时用 AskUserQuestion 单点确认（不连发多选）。"很低"对应"较大"对齐成同一数值 20 → 数据自然两段切分（"已完成<20 红 / 已取消>20 黄"），且抓数能反向验证（红 0 单 = 已完成实付下限≥20，证明阈值合理）。

6. **崇礼脚本幂等重跑可跳过验证**：把脚本参数化（常量改函数入参）只改形参不改主逻辑，等价确定无需重测。但若目标文件 Excel 还开着会 PermissionError（写 xlsx 锁），先 `ls '~$<同名>.xlsx'` 探锁，或换 AskUserQuestion 让用户关 Excel。

7. **end-work 不再需要确认（用户拍板）**：触发 end-work 后直接落盘 CLAUDE.md + sessions/ + `git commit + push`，**永远不需要 AskUserQuestion 询问**。之前 SKILL.md 第 3 步「Show the user what will be added / Ask for confirmation before updating」被用户作废；记入 auto-memory 保证跨会话/跨机持久。

8. **南山雪票存在大量"未还卡"订单**（实施后取消的押金不平判定揭示）：85 单押金合计 > 退款合计（典型场景：5 张票全没还卡，押金 ¥2000 退款 ¥0）。虽然这次没把它做成可视化标记，但业务上是需关注的数据特性—未来真要追退押金，按 `Σ ski_pass.deposit − Σ ski_pass.refund_amount ≥ 0.01 AND order.paid > 0` 筛订单即可（不必脚本支持）。
